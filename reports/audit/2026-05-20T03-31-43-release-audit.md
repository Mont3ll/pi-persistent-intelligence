## Code Architecture Findings

### Strengths
- **Types are comprehensive and coherent** ‚Äî `src/types.ts` covers all lifecycle stages with backward-compatible optional fields
- **Trust pipeline is clean** ‚Äî `trust.ts` is single-responsibility; `isAutoApplyEligibleCandidate(candidate, mode)` is readable
- **Processor pipeline is testable** ‚Äî `ProcessorTrace` makes exclusion reasons auditable
- **Patch boundary hardening is effective** ‚Äî `addMemoryRecord()` throws without `PatchApplyContext`; internal flows use `addMemoryRecordFromPatch()`
- **Evidence/tombstone semantics are sound** ‚Äî privacy purge removes content completely; audit_preserving keeps for audit trail
- **Meta-consolidation is safely designed** ‚Äî no auto-apply path; report-only output

### Risks
- **`unsafeAddMemoryRecord`, `unsafeReplaceLayerRecords`, etc. exported from `src/store.ts`** ‚Äî these are accessible to any consumer who imports from `src/store.ts`. Not in public `index.ts` exports, but reachable. Acceptable risk for 0.x release; internal consumers (tests, migration) need them.
- **`src/correction.ts` phrase "prefer X before Y" not matched** ‚Äî "prefer bun test before typecheck" is a real correction pattern. Borderline case. See Finding F-01.
- **`applyPatch()` vs `applyPatchAndSync()` caller responsibility** ‚Äî the distinction is documented and tested; `/apply-memory-patch` command correctly routes delete ops to `applyPatchAndSync()`, but other flows calling `applyPatch()` rely on caller to sync. This is acceptable but could surprise future contributors.
- **`meta-consolidation` `clusterL2Records` uses `normalized_key` first, then falls back to `profile_id|ruleType`** ‚Äî records without `normalized_key` group by ruleType only, which may produce overly broad clusters for legacy records. Low risk at 0.x.

### Questionable areas
- **`loadLayerRecords` in `store.ts` does NOT filter tombstoned records** ‚Äî `loadLayerRecords` returns all records without tombstone check (unlike `loadActiveRecords`). This is intentional (audit trail), but callers using `loadLayerRecords` directly could see deleted records. The distinction should be documented.
- **`session-end auto-curation` in `index.ts`** ‚Äî the `session_shutdown` auto-curation block calls `curateInbox()` and `applyPatch()` but does NOT call `syncFtsIndex()` before qmd update. The final `syncFtsIndex()` call happens inside the `session_shutdown` via `await updateQmd(); syncFtsIndex(root, ftsIndex)` ‚Äî but only inside the `consolidationResult` block at the bottom. The auto-curation block at lines 394‚Äì428 calls `applyPatch` then logs but does NOT call `syncFtsIndex` itself. The FTS is updated later at line 439+. This is technically correct (session end is not a user-facing immediate operation) but worth noting.

### Recommended refactors (deferred)
- Add a code comment to `loadLayerRecords` noting it does not filter tombstones
- Consider prefixing `unsafe*` exports with `_internal` or moving them to a separate module
- Add `syncFtsIndex` call inside session-end auto-curation block for consistency

## Findings

### F-01 ‚Äî Correction detection: "prefer X before Y" pattern not matched
- **Severity**: INFO
- **Affected files**: `src/corrections.ts`
- **Description**: The phrase "prefer bun test before typecheck" is a natural correction expression but is not matched by `maybeCorrectionSignal()`. The detector requires "prefer X **over/to/instead of** Y" ‚Äî the pattern `/\bprefer\s+.+\s+(?:over|to|instead\s+of)\b/i`. "Before" as a comparison operator is not included.
- **Reproduction**: `maybeCorrectionSignal("Going forward, prefer bun test before typecheck.")` ‚Üí `false`
- **Note**: The detector is intentionally conservative. "before" can be temporal ("before pushing") not comparative. Adding it would increase false positives. The dogfood checklist uses a correctly-detectable phrase.
- **Recommended fix**: Add a note to `src/corrections.ts` documenting this edge case. No code change needed.
- **Fixed during audit**: No (INFO level, intentional behavior)

### F-02 ‚Äî `loadLayerRecords` returns tombstoned records
- **Severity**: LOW
- **Affected files**: `src/store.ts` (line 21)
- **Description**: `loadLayerRecords(root, "L2")` does not filter tombstoned records. `loadActiveRecords()` does. Callers who use `loadLayerRecords` directly (including tests, `src/maintainer.ts`, internal render flows) can see `status: "deleted"` records.
- **Reproduction**: Privacy-purge a record, then call `loadLayerRecords` ‚Äî the `status: "deleted"` record is returned.
- **Context**: This is intentional for audit trail purposes. The `render.ts` already filters `status: "deleted"`. The distinction is correct but undocumented.
- **Recommended fix**: Add JSDoc comment to `loadLayerRecords` explaining it returns all records including deleted (for audit trail). `loadActiveRecords` is the correct API for injection/search.
- **Fixed during audit**: Yes (comment added below)

### F-03 ‚Äî Session-end auto-curation does not immediately sync FTS
- **Severity**: LOW
- **Affected files**: `index.ts` session_shutdown handler
- **Description**: The auto-curation block in `session_shutdown` calls `applyPatch()` but the FTS sync happens only in a separate code block at session end. If the session ends between auto-curation and the final FTS sync call (e.g., error during consolidation), FTS could be briefly stale. This is a session-boundary edge case, not a user-facing runtime issue.
- **Recommended fix**: Add `syncFtsIndex(root, ftsIndex)` immediately after the auto-curation `applyPatch()` call. Small change.
- **Fixed during audit**: Yes (applied below)

### F-04 ‚Äî `setup-session-search` command undocumented
- **Severity**: INFO
- **Affected files**: `index.ts` (line 833), `README.md`
- **Description**: `/setup-session-search` is registered but not in the README commands table. It shows session index info and qmd embed hint.
- **Recommended fix**: Add to README as a debug utility or leave undocumented (acceptable for internal tooling).
- **Fixed during audit**: No (INFO, acceptable)
